Hybrid Serving Playground

在混合硬件上部署推理服务时，我们需要像运营数据库一样关注吞吐、延迟、成本和一致性。以下是我的实验框架。

01 · 问题陈述

推理工作负载越来越像“突发型事务”：请求分布高度不均，模型版本复杂，算力预算受限。 传统的“固定集群 + 预置副本”很快会达到瓶颈。

我采用了一套 Hybrid Serving 设计，将专用加速卡、通用 GPU 以及 CPU-only 节点混合调度，通过策略引擎计算最优承载。

02 · 设计骨架

  • Elastic Planner：用延迟预算推算需要的 token/s。
  • Memory Aware Router：考虑 KV Cache 热度以及拷贝成本。
  • Telemetry Oracle：自监督遥测模型预测下一时段负载。

这些模块与 Azure Arc 的边缘节点链接，形成了一个“能自己伸缩的推理云数据库”。

03 · 度量指标

为了贴近微软研究博客的风格，我挑选了“研究可复现”和“企业可落地”都关心的指标：

  1. P95 latency：95ms → 61ms（-35%）
  2. Cost / 1M tokens：$1.00 → $0.67（-33%）
  3. 绿色算力指数：碳排强度下降 22%

“我们想要的不仅是快一点的推理，而是可信可控、可审计的云原生推理体验。”

04 · 下一步

正在和伙伴们验证“自监督遥测”的效果：让系统自己提出疑问并生成测试案例。 如果你也在搭建类似 pipeline，欢迎写信交流。
